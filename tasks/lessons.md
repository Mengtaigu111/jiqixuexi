# Lessons Learned

## Rules

- 当用户要求“等卡空了再跑”或“每隔一段时间看卡是否空闲”时，不要改成 CPU full 实验；应停止不符合要求的本地实验，使用 tmux 启动 watcher 轮询 GPU，空闲后再用 GPU 跑正式实验。
- 当用户明确说“不准 block”并要求定期回来检查时，不要再次把 goal 标记为 blocked；应保持会话继续巡检，按用户指定间隔汇报 GPU、tmux、日志和结果目录状态，直到 full 实验完成。
- 当用户要求第三个改进模型用于课程结果展示时，不能只满足“结构新颖”；最终对比表里第三模型应成为最优或至少按用户目标继续优化到最优。若单体模型不够强，应考虑验证集加权集成、stacking、校准等严格不使用测试集调参的方法，而不是接受次优结果。
- 当用户询问“内存”并明确纠正“不是卡，是服务器内存”时，应使用 `free -h` 和进程 RSS 检查系统 RAM，不要继续汇报显存；同时说明 Linux `buff/cache` 是可回收缓存，不等同于内存被任务占满。
- 当用户要求课程报告“多点分析、多放分析图片”时，不能只把结果表和基础预测曲线塞进 Word；报告正文必须增加过程图、改进幅度图、逐 seed 稳定性图、误差分布图、逐预测步误差图、校准/专家选择图，并配套解释这些图回答“为什么第三模型变好、和前两个相比如何、仍有哪些限制”。
- 当用户继续要求“Word 格式再改进、好好排版”时，不能只生成内容正确的 `.docx`；必须把封面、静态目录层级、H1/H2 视觉层级、表格表头/斑马纹、图宽/图注、页眉页脚和 live 页码字段固化到生成脚本中，并用 `officecli validate`、outline/stats/issues、图片 alt、token 泄漏和 HTML 预览共同验证。
- 当用户追问缺失值“补 0 是否合理”时，应明确区分核心功率/电压列与缺失的可选天气列：核心连续传感器数据不应把缺测直接解释为真实 0，项目当前代码的 `fillna(0)` 主要是前向/后向填充后的兜底；报告中应表述为先 `? -> NaN`，再时间序列前后向填充，剩余不可恢复字段才兜底置 0。
- 当用户要求“参考别的文献，不要只看给你的文献”时，应主动查缺失数据统计/时间序列插补/电力负荷预测预处理相关资料；对连续用电功率列，不能建议直接 zero imputation，除非 0 有真实物理含义或作为缺失 mask 的伴随编码。
- 当用户指出报告“模型数量、表格和结论不一致”时，不能只改一句结论；必须统一 README、Markdown 草稿、PDF/Word 生成脚本、结果表和图表的口径，并让报告表格直接从 `results/metrics/summary.csv` 格式化生成，避免手工复制产生三模型/四模型或 30/40 次实验矛盾。
- 当用户纠正“正式报告应该是 3 个模型”时，不能把中间消融模型继续放在主结果表里；正式 `summary.csv`、主图、提交检查和封面实验次数都必须只统计 LSTM、Transformer、最终 DMSAFormer，HybridTCNTransformer 只能作为改进过程/消融说明。
- 当用户指出最终报告仍有“DMSAFormer 定位不清、validation 校准风险、作者贡献占位、PDF 断词和参考文献格式”问题时，应优先修正式报告表达和生成脚本：明确 DMSAFormer 是最终提出模型、Hybrid 是中间改进/消融对照，写清 test set 不参与选择或校准，并消除“待填写”等提交占位符。
- 当用户指出改进模型“不是惊艳型创新、容易显得拼装”时，报告不能夸大为全新架构；应改写为面向任务的工程组合型改进，明确模块动机，并用已有 v1/v2/final 演进结果或模块作用表做消融说明。没有实际重跑的 ablation 数字不能编造。
- 当审查发现 DMSAFormer 最终结果实际来自 baseline 预测改名时，必须优先修正实验诚信边界：最终 DMSAFormer 输出只能来自 DMSAFormer 自身 checkpoint，validation 校准只能拟合自身验证集预测；报告应诚实呈现 90 天不最优、365 天最优的结果，不能为了榜单数字牺牲方法真实性。
